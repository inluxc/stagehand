# Requirements Document

## Introduction

This feature adds two CLI commands to the Playwright Framework Template: `npx playwright-framework init` for scaffolding new projects and `npx playwright-framework add` for adding fixtures to existing projects. These commands reduce manual setup effort and ensure consistent project structure across teams.

## Glossary

- **CLI**: The command-line interface entry point exposed via the package's `bin` field in `package.json`
- **Init_Command**: The CLI subcommand that scaffolds a new project with the framework's directory structure, configuration files, and selected fixtures
- **Add_Command**: The CLI subcommand that adds a fixture (and its dependencies) to an existing project
- **Fixture**: A reusable test resource (database, kafka, mobilewright, openapi, redis) that follows the setup/use/teardown lifecycle pattern
- **Fixture_Registry**: The `src/fixtures/index.ts` file that composes all fixture definitions into the extended Playwright test object
- **Environments_File**: The `environments.json` configuration file that defines per-environment settings for each fixture
- **Scaffold**: The set of files and directories generated by the Init_Command to create a working project
- **Project_Root**: The target directory where the CLI writes generated files

## Requirements

### Requirement 1: Project Initialization

**User Story:** As a developer, I want to scaffold a new Playwright Framework project with a single command, so that I can start writing tests without manual file setup.

#### Acceptance Criteria

1. WHEN the user runs the Init_Command, THE CLI SHALL prompt the user with a multi-select list of available fixtures (database, kafka, mobilewright, openapi, redis) where all fixtures are deselected by default
2. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `package.json` file containing only the dependencies required by the selected fixtures
3. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `tsconfig.json` file with TypeScript compiler options matching the framework's conventions
4. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `playwright.config.ts` file with a default Playwright configuration
5. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate an `environments.json` file with placeholder configuration for each selected fixture in the `local` environment
6. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `.env.local.example` file listing the environment variables relevant to the selected fixtures
7. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `src/fixtures/index.ts` Fixture_Registry file that imports and composes only the selected fixtures using Playwright's `test.extend()`
8. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate individual fixture files under `src/fixtures/` for each selected fixture, named `<fixture>.fixture.ts`
9. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `src/config/` directory containing the env-loader, schema, and loader modules
10. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `src/errors.ts` file with the framework error classes
11. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate a `src/index.ts` barrel file that re-exports the selected fixtures and config modules
12. WHEN the user confirms fixture selection with at least one fixture selected, THE Init_Command SHALL generate an example test file under `tests/examples/` for each selected fixture, named `<fixture>.spec.ts`
13. IF the current working directory already contains a `package.json` file, THEN THE Init_Command SHALL abort without generating any files and display an error message indicating that a project already exists in this directory
14. WHEN scaffolding completes successfully, THE Init_Command SHALL print a summary listing each generated file path and the next steps: install dependencies with the package manager and configure environment variables
15. IF the user confirms fixture selection with zero fixtures selected, THEN THE Init_Command SHALL abort without generating any files and display an error message indicating that at least one fixture must be selected
16. IF any file cannot be written to the current working directory during scaffolding, THEN THE Init_Command SHALL abort, remove any files already generated in the current run, and display an error message indicating which file failed to write

### Requirement 2: Add Fixture to Existing Project

**User Story:** As a developer, I want to add a new fixture to my existing project with a single command, so that I can extend my test capabilities without manually wiring up files.

#### Acceptance Criteria

1. WHEN the user runs the Add_Command with a fixture name argument, THE CLI SHALL validate that the fixture name is one of the supported fixtures (database, kafka, mobilewright, openapi, redis) using case-insensitive comparison
2. IF the user provides an unsupported fixture name, THEN THE Add_Command SHALL exit with a non-zero exit code and print an error message that lists all valid fixture names
3. IF the Add_Command is run in a directory that does not contain the expected project structure (missing `src/fixtures/` directory, `package.json`, or `environments.json`), THEN THE Add_Command SHALL exit with a non-zero exit code and print an error message indicating the missing project structure
4. WHEN a valid fixture name is provided, THE Add_Command SHALL generate the fixture file at `src/fixtures/{name}.fixture.ts`
5. WHEN a valid fixture name is provided, THE Add_Command SHALL update the Fixture_Registry at `src/fixtures/index.ts` to add an import statement for the new fixture module and spread its exported fixture object into the composed fixtures
6. WHEN a valid fixture name is provided, THE Add_Command SHALL add the fixture's required dependencies to the `dependencies` section of `package.json` with pinned major versions
7. WHEN a valid fixture name is provided, THE Add_Command SHALL add a configuration entry for the fixture to the `local` environment in `environments.json` containing the fixture's required configuration keys with empty or default values
8. WHEN a valid fixture name is provided, THE Add_Command SHALL generate an example test file at `tests/examples/{name}.spec.ts`
9. IF any target file (fixture file at `src/fixtures/{name}.fixture.ts` or example test at `tests/examples/{name}.spec.ts`) already exists at its target path, THEN THE Add_Command SHALL skip generation of that file and print a warning message identifying the skipped file path
10. IF the fixture entry already exists in the Fixture_Registry import list, THEN THE Add_Command SHALL skip the registry update and print a warning message indicating the fixture is already registered
11. WHEN the add operation completes successfully, THE Add_Command SHALL print a summary listing each file created or modified and print a reminder to run the package manager install command to fetch newly added dependencies

### Requirement 3: Fixture Dependency Resolution

**User Story:** As a developer, I want the CLI to automatically include fixture dependencies when I add a fixture, so that I do not encounter runtime errors from missing dependencies.

#### Acceptance Criteria

1. WHEN the user adds the redis fixture, THE Add_Command SHALL add the `redisConfig` fixture definition to the Fixture_Registry file (`src/fixtures/index.ts`) including its import statement and composition in the `allFixtures` object, so that the `redisClient` fixture can resolve its `redisConfig` parameter at runtime
2. WHEN the user adds the mobilewright fixture, THE Add_Command SHALL add both the `mobilewrightDevice` and `mobilewrightScreen` fixture entries to the Fixture_Registry file including their import statements and composition in the `allFixtures` object
3. THE CLI SHALL maintain a hardcoded dependency map that defines which npm packages each fixture requires (openapi: openapi-client-axios; database: pg, mysql2, better-sqlite3; kafka: kafkajs; redis: ioredis; mobilewright: mobilewright, @mobilewright/test)
4. WHEN adding a fixture, THE Add_Command SHALL add all npm packages from the dependency map for that fixture to the project's `package.json` dependencies field with pinned major-version ranges (e.g., `^X.Y.Z`)
5. IF a fixture dependency is already present in the Fixture_Registry, THEN THE Add_Command SHALL skip adding that dependency and not produce duplicate entries
6. IF the Add_Command fails to write dependency changes to `package.json` or the Fixture_Registry file, THEN THE Add_Command SHALL exit with an error message indicating which file could not be updated and leave the file in its original state

### Requirement 4: Interactive Prompts

**User Story:** As a developer, I want the CLI to guide me through setup with interactive prompts, so that I can make informed choices without memorizing command flags.

#### Acceptance Criteria

1. WHEN the Init_Command is run without the `--fixtures` flag, THE CLI SHALL display an interactive multi-select prompt listing all supported fixtures (database, kafka, mobilewright, openapi, redis) with none pre-selected
2. WHEN the Init_Command is run with a `--fixtures` flag containing a comma-separated list, THE CLI SHALL skip the interactive fixture prompt and use the provided fixture names
3. IF the `--fixtures` flag contains a fixture name not in the supported set (database, kafka, mobilewright, openapi, redis), THEN THE CLI SHALL exit with a non-zero exit code and an error message listing the valid fixture names
4. WHEN the Add_Command is run without a fixture name argument, THE CLI SHALL display an interactive single-select prompt listing all supported fixtures (database, kafka, mobilewright, openapi, redis)
5. WHEN the `--yes` flag is provided with the Init_Command without a `--fixtures` flag, THE CLI SHALL skip all interactive prompts and select all supported fixtures as the default selection
6. IF the CLI detects that stdin is not a TTY and no `--yes` flag is provided and required prompts cannot be displayed, THEN THE CLI SHALL exit with a non-zero exit code and an error message indicating that interactive mode requires a terminal or the `--yes` flag

### Requirement 5: Secrets Provider Selection

**User Story:** As a developer, I want to choose a secrets provider during project initialization, so that my project is configured for the correct secret management approach from the start.

#### Acceptance Criteria

1. WHEN the Init_Command runs, THE CLI SHALL prompt the user to select exactly one secrets provider from the list: aws, azure, env-file, gitlab, vault, with "env-file" pre-selected as the default option
2. WHEN the user selects a secrets provider, THE Init_Command SHALL include a `secrets` object in each environment entry of the generated `environments.json` file containing at minimum the `provider` field set to the selected provider name and an empty `keyMappings` object
3. WHEN the user selects a secrets provider, THE Init_Command SHALL include environment variable placeholders in the generated `.env.local.example` file that correspond to the selected provider's required connection parameters (e.g., region and secret prefix for aws, project ID and API URL for gitlab, vault URL and mount path for vault, key vault URL for azure)
4. WHEN the Init_Command is run with a `--secrets-provider` flag, THE CLI SHALL skip the secrets provider prompt and use the provided value
5. IF the `--secrets-provider` flag contains a value that does not case-insensitively match one of aws, azure, env-file, gitlab, or vault, THEN THE CLI SHALL exit with a non-zero exit code and display an error message indicating the invalid value and listing the valid provider names
6. IF the user dismisses the secrets provider prompt without making a selection, THEN THE Init_Command SHALL use "env-file" as the default secrets provider

### Requirement 6: CLI Entry Point and Help

**User Story:** As a developer, I want clear help output and discoverable commands, so that I can use the CLI without consulting external documentation.

#### Acceptance Criteria

1. THE CLI SHALL be invocable via `npx playwright-framework init` and `npx playwright-framework add`
2. WHEN the user runs the CLI with `--help`, THE CLI SHALL display usage information listing all available commands and their descriptions and exit with code 0
3. WHEN the user runs `init --help`, THE CLI SHALL display the Init_Command options including `--fixtures`, `--secrets-provider`, and `--yes` and exit with code 0
4. WHEN the user runs `add --help`, THE CLI SHALL display the Add_Command usage including the fixture name argument and `--yes` flag and exit with code 0
5. IF the user runs the CLI with a command that is not `init` or `add`, THEN THE CLI SHALL exit with a non-zero exit code and display an error message indicating the unrecognized command name followed by the help text
6. WHEN the user invokes the CLI with `--version`, THE CLI SHALL display a version number matching the package.json `version` field and exit with code 0
